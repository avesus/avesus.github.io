Frontend/Backend Page-State Architecture
May 19, 2023

The backend should be able to answer the screen the user is actually on.

This note describes a single-page frontend backed by page-shaped business state: current data, permissions, available actions, navigation targets, and workflow context returned together instead of reconstructed from scattered generic resources.

Architecture noteOriginally posted as a 2023 project-status update; rewritten here around the actual system pattern.

Included: Vue/Quasar/Vue-router frontend context, JSON backend responses, page-state requests, an order-list screen example, permissions, actions, workflow context, and where the pattern should not be used.

Boundary: this is an architecture note, not a finished framework or a replacement for stable external APIs.

The Problem

Modern frontend applications are good at interaction: transitions, drawers, validation, routing, and local UI state. The backend usually still owns the business rules, data joins, permissions, and workflow rules.

The awkward part is the gap between them. A frontend can end up rebuilding one screen from many resource calls, local cache merges, permission checks, and inferred available actions. The screen may look modern while the business state behind it becomes scattered and easy to stale.

Page-Shaped Business State

The alternative in this note is not backend-rendered HTML. The frontend still renders the interface with Vue, Quasar, and Vue-router. The backend sends JSON.

The difference is the shape of the request and response. The frontend tells the backend which page, panel, list, detail view, or workflow step it needs. The backend returns one coherent state packet for that visible screen.

Order List Example

For an order-list screen, the page-state response can carry the rows to show, the selected filters, pagination, user-visible status labels, permission flags, available actions, detail-panel links, and warnings that affect the next user action.

The frontend does not have to discover all of that by stitching together generic order resources, user resources, permission resources, and action endpoints. It receives the business state for the screen and renders it.

Backend Responsibility

The backend owns the parts that should not be guessed in the browser: authorization, workflow rules, current availability of actions, joins across business data, and the state transition rules behind buttons or links.

That does not make the backend a template renderer. It makes the backend responsible for returning the business shape the frontend is allowed to show now.

Frontend Responsibility

The frontend owns rendering, interaction, transitions, layout, local form editing, and the ergonomics of moving between views. It should not have to rediscover business rules by coordinating a pile of unrelated responses.

This keeps the frontend expressive without forcing it to become a second backend written in browser code.

Failure Mode Avoided

The failure mode is a screen assembled from too many partial truths: cached data from one request, permission state from another, current workflow state from a third, and available actions inferred from stale local assumptions.

A page-state response reduces that class of bug by making the backend answer one question: what can this user see and do on this screen right now?

Where Not To Use It

This pattern is a poor fit for public integration APIs where external callers need stable resource contracts. It is also a poor fit when many independent clients need the same reusable entity model, or when the client must work offline for long periods and reconcile changes later.

It fits best when the product workflow matters more than a generic resource ceremony: internal tools, business dashboards, operational screens, admin panels, approval flows, and systems where permissions and available actions are central to the page.

Original Motivation

The original 2023 comparison was to PHP only in one limited sense: old server pages could ask for exactly the data the page needed. The useful part was direct access to page-specific business state, not mixing SQL and HTML or returning to server-rendered interfaces.

The cleaned-up architecture keeps the modern frontend and gives the backend a better job: answer the current screen with the data and actions that belong to that screen.
