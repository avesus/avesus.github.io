When a Computer-Architecture Idea Needs More Than a Diagram

A research proposal can describe a new fabric, data path, configuration method, or hardware-software boundary without yet answering the questions that make the idea testable. What is the machine state? Which transitions are local? How is configuration delivered? What must remain independent between the reference model and the RTL? Which FPGA result would reduce the next technical risk?

That gap is a good work package for me. I can make the proposed mechanism explicit, build a runnable reference model, implement an independent SystemVerilog version, add repeatable verification, and define the synthesis, place-and-route, or board measurements that should follow. The result is not a decorative demo. It is an artifact another engineer can run, challenge, and extend.

This work is especially useful when a principal investigator or R&D lead has the scientific question and institutional program, but needs one person who can move coherently between architecture, browser-GPU modeling, RTL, verification, FPGA tooling, and a readable public or internal handoff.

Cartilage: A Reconfigurable Fabric With Inspectable State and Routing

Cartilage Core is the current compact proof package behind this offer. It is a nearest-neighbor reconfigurable fabric in which local role, routing, ownership, and configuration are part of the inspectable machine rather than hidden behind a fixed API.

The browser model installs 36 seven-bit records into a child-owned 6x6 region: 252 payload bits followed by one apply pulse. An interior multiplexer, a local ground cell, and directional cells implement and route a checked AND circuit. The independent SystemVerilog layer has no fabric-wide application clock; application and overlay signals remain continuous local levels while configuration state changes on routed local edges.

Established: executable WebGL behavior, canonical SystemVerilog checks, exact serial installation, differential checks for the flattened iCE40 encoding, and a recorded routed physical-tile result.

Boundary: architecture and FPGA implementation evidence, not board execution, timing closure, measured frequency or power, fabricated silicon, or a general compiler and placer/router.

From WebGL Reference Model to SystemVerilog RTL and an FPGA Route

The useful part of Cartilage is the evidence chain. Its WebGL1 model makes a large local state machine visible and directly interactive. The canonical SystemVerilog modules state the hardware contract independently. A self-checking Verilator testbench exercises primitives, rotated port priorities, local DFF edges, the route-shadow handoff, all 36 installed records, and all four AND truth-table rows.

The public repository now also records a completed Yosys synthesis, nextpnr packing/place-and-route, and icepack flow for a fully edge-bonded 9x8 Cartilage tile on an iCE40HX8K-CT256. Under that particular APP/C/D perimeter-interface and package-pin contract, the 72-cell result uses 5,513 LUT4, 1,512 DFF, 6,811 of 7,680 packed logic cells, and 205 of 206 bonded user I/O. The machine-readable result and reproduction flow are published with the source.

That route is a concrete capacity and connectivity result. It does not claim that the browser AND circuit ran on a board, and it does not substitute for physical-clock analysis, measured Fmax, power, thermal behavior, or multi-device operation. Those are exactly the kinds of next measurements a well-scoped research collaboration can define.

What a Research Collaboration Can Produce

A useful engagement can be small and sharply bounded. It might produce one or several of these connected artifacts:

Architecture
An explicit state, timing, routing, configuration, and interface contract with the hard assumptions exposed.
Reference model
An executable model that makes the proposed mechanism inspectable and gives the research team a comparison oracle.
RTL
Independent SystemVerilog or Verilog modules designed around the real hardware boundary rather than copied from the model.
Verification
Self-checking tests, differential comparisons, exact vectors, waveforms, and a concise statement of what passed and what remains open.
FPGA prototype
Synthesis and place-and-route evidence, resource and timing reports, a board plan, or same-target comparisons that answer the next program risk.
Handoff
Source, reproduction commands, diagrams, evidence notes, and defined follow-on work that a larger team can extend.

The same approach applies beyond Cartilage. The FPGA systems evidence map includes RF paths, network RTL, adjacent-device reconfiguration, and a bubbles-free bit-serial multiplier whose next step is a same-target RTL and FPGA comparison.

Where This Fits Advanced Computing and Microelectronics R&D

The public language used by major U.S. research programs is a useful map of the teams this work can support. The National Science Foundation groups related work under Computer Systems Research and Software and Hardware Foundations. Its technology topics include computational architecture, processor architecture and design, and instrumentation and hardware systems. NIST describes CHIPS research and development as work to invent, develop, prototype, and deploy foundational semiconductor technologies.

Those categories become concrete work here: computer-architecture prototyping, reconfigurable computing research, hardware-software co-design, SystemVerilog RTL, Verilator verification, FPGA implementation, design-space exploration, and technology-transition artifacts.

Cartilage is also not a relabeling of an established FPGA feature. For example, AMD Dynamic Function eXchange modifies designated sections of an FPGA while the remainder continues to operate. Cartilage instead investigates local roles, ownership, edge routing, and configuration transport inside the fabric model. That distinction is a research question, not a product-comparison claim.
