The reason I care about this formal page is practical: if a substrate can be described with a small kit of Boolean pieces, then the hidden part becomes easier to inspect and harder to mystify.

Wires, constants, 2:1 multiplexers, feedback, configuration registers, and unbounded tiling are enough to define a universal, reconfigurable Boolean substrate.

That is the engineering claim. The paper makes the construction explicit so the reader can treat computation as a clocked next-state function over bits, not as a collection of unrelated machine metaphors.

What is preserved here: the Zenodo DOI, PDF, license, citation metadata, and the primitive set: wires, constants, 2:1 MUXes, feedback, configuration registers, and unbounded tiling.

What this does not prove: that a physically local maker-scale active-device process exists. Boolean sufficiency and physical fabrication are different problems.

What The Paper Gives

The paper defines a minimal computation model whose primitives are ordinary Boolean hardware primitives: wires, constants, and 2:1 multiplexers. With feedback and registers, those primitives become sequential switching networks. With configuration registers, the same substrate can be programmed. With unbounded tiling, it has unbounded memory and therefore universal computational power.

The point is not that multiplexers are mystical. The point is that a MUX fabric gives a small implementation path: conditional selection, state, reconfiguration, routing discipline, and replication can all be expressed inside the same Boolean system.

Why MUXes Are Enough

Constants supply fixed zeros and ones. Wires move bits. A 2:1 MUX selects one of two inputs under control of a select bit. Networks of those MUXes implement arbitrary Boolean functions.

Once outputs feed registers and registers feed later logic, the fabric becomes a clocked state machine. Once some registers are treated as configuration, structure becomes data. Once the pattern can be tiled without a fixed global bound, memory and machine size are no longer fixed by the first drawing.

Implementation Path

Read this as an implementation model for a reconfigurable substrate. The important terms are local wires, constants, 2:1 MUXes, feedback, configuration state, tiling, and the protocol that installs new configuration. The paper is the formal bridge; it says why that small kit is sufficient.

It also explains why reconfiguration should not be treated as outside the machine. Programming a fabric is itself a Boolean, clocked process. Configuration bits move through a substrate, land in registers, and change the next-state function the same way any other state changes it.

Relation To Cartilage

Cartilage is the live implementation direction attached to this paper. The primary Cartilage demo shows a tiled browser/GPU fabric where a local reconfiguration port streams bits into bounded regions and replaces tile roles such as wires, MUXes, constants, intersections, and further reconfiguration ports.

The Cartilage nested-instantiation article is the next useful read because it turns the formal claim into an inspectable artifact: module boundaries, local ports, serial configuration streams, bounded regions, composability, and no global magic controller.

Tradeoff

The construction is intentionally small. It does not start from a convenient CPU, an HDL standard library, or a friendly programming language. It starts from the fewest Boolean pieces that still reach state, configuration, and unbounded growth.

The tradeoff is that the first description looks lower-level than the machines people usually name. The gain is that every layer has a direct substrate interpretation.

Cartilage Visual Language

The 32 Cartilage cell-role codes are a compact body-type alphabet: reconfiguration port, cross, constants, four wire orientations, and six MUX modes in four orientations. The Cartilage Visual Language article shows the rendered key and decodes all 32 codes.
